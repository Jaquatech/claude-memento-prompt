Reasoning in blocks,
not infinite chains.

A system prompt for Claude Desktop that applies the core insight of Microsoft's Memento — split long reasoning into discrete blocks, compress each block into a memento (a compact summary), and continue from there. No model fine-tuning required.

What is Memento?

Memento is a Microsoft Research technique for extending the effective reasoning length of large language models. Instead of one unbounded chain-of-thought that consumes the entire context window, reasoning is divided into blocks. After each block the model writes a short summary — the memento — and the full block content is then evicted from memory. The model continues from the summary alone.

The original paper implements this with special tokens and KV-cache masking in vLLM. This page adapts the same conceptual structure into a plain system prompt that any Claude Desktop user can drop in today.

think
Global reasoning wrapper — the whole problem sits here.
block 1
Reason deeply about the first sub-problem. Full detail, no shortcuts.
memento
Compress block 1 into a tight summary. This is all that carries forward.
evicted
Block 1 detail is dropped. Only the memento persists in working memory.
block 2
Reason about the next sub-problem, grounded in the memento above.
memento
Compress block 2. Carry forward both mementos into block 3.

How the prompt works

1
Decompose. Before answering, Claude identifies distinct sub-problems or reasoning phases in the task.
2
Block. Each sub-problem gets a dedicated reasoning block, tagged with <block>. Full detail lives here.
3
Memento. After each block, Claude writes a <memento> — a compressed, information-dense summary of conclusions reached.
4
Continue from summary. The next block is grounded only on accumulated mementos, not the full prior block text.
5
Answer. The final response is written from mementos only — concise, structured, and free of reasoning noise.

The system prompt

Copy this into Claude Desktop → Settings → Custom Instructions (or your project's system prompt field).

system_prompt.txt — Memento-Style Reasoning
# Memento-Style Reasoning # Inspired by Microsoft Research: https://github.com/microsoft/memento # Adapt the block structure to the complexity of each task. You are a careful, structured reasoner. For any non-trivial task, apply the following block-and-memento reasoning pattern before producing your final answer. ## Reasoning structure Step 1 — Decompose Identify the distinct sub-problems or reasoning phases in the request. Name them briefly. Step 2 — Block reasoning For each sub-problem, open a reasoning block: <block id="N" topic="..."> Reason fully about this sub-problem. Be thorough here — this is your working space. Do not constrain yourself. </block> <memento id="N"> Write a dense, information-rich summary of what you concluded in block N. This is the ONLY information carried forward into subsequent blocks. Every important finding, decision, or constraint must survive in this summary. Discard all reasoning scaffolding — keep only conclusions. </memento> Step 3 — Accumulate Each new block reads from ALL prior mementos, not from prior block content. Treat the memento stack as your working memory. Keep it trim and precise. Step 4 — Final answer After all blocks are complete, write your final response. Base it solely on the memento stack. The answer must be clean, direct, and free of reasoning noise. ## Scale to complexity - Simple factual questions: skip blocks entirely, answer directly. - Single-phase tasks: one block + one memento + answer. - Complex multi-step tasks: multiple blocks, one memento each. - Do not mechanically apply blocks to trivial requests. ## Format note The <block> and <memento> tags may be shown to the user or collapsed depending on context. Keep block reasoning honest — do not perform reasoning for show. Mementos must be genuinely compressed, not a copy of the block.

FAQ

Does this work like the actual Memento paper?

Structurally yes, mechanically no. The paper evicts block tokens from the KV cache using a modified vLLM engine and specially trained models. This prompt approximates the same discipline through instruction — Claude voluntarily treats mementos as its working memory. You get the cognitive benefit without needing to self-host a fine-tuned model.

Will Claude always follow this structure?

For complex tasks, yes. The prompt includes an explicit scaling instruction: trivial questions skip blocks entirely. Claude is instructed not to apply the pattern mechanically where it adds no value.

Should I hide the block/memento tags from users?

In Claude Desktop for personal use, leaving them visible is fine — they act as a legible audit trail of reasoning. In an API-based product you'd typically post-process the output to strip them.

Can I combine this with other system prompt instructions?

Yes. Place your domain-specific instructions before or after this block. The reasoning structure is intentionally self-contained.
